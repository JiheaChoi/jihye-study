## Problem Statement

항공사 IT 인프라 조직에서 해외 각 지점이 발송하는 시스템 장애/오류 메일이
체계 없이 처리되고 있다. 메일이 묻히거나 담당자 확인이 늦어지면서 장애 대응이
지연된다. 지점 × 시스템 유형의 복합 조건에 따라 담당자가 다르지만 현재 이 라우팅을
자동화하는 수단이 없다.

## Demand Evidence

- 현재 프로세스 없음 (비정형 — 그냥 때마다 다름)
- 메일 누락으로 인한 장애 대응 지연이 실제로 발생 중
- 항공사 시스템 장애는 운항 리스크와 직결 — 지연 비용이 높음
- 담당자 본인이 이 문제를 직접 경험하며 해결책을 설계 중

## Status Quo

- 해외지점 장애 메일은 특정 메일박스로 수신
- 수신 후 수동으로 읽고 담당자에게 개인적으로 전달
- 일관된 프로세스 없음 — 누가 확인할지, 언제 확인할지 정해진 기준 없음
- Slack 알림 없음. 담당자가 자리를 비우면 다음날 알게 되는 경우 있음

## Target User & Narrowest Wedge

**타겟 유저:** 항공사 IT 인프라 조직 내 시스템/지점별 담당 엔지니어.
지점(유럽, 미주, 동남아 등) × 시스템(ERP, 네트워크, 서버 등) 조합으로 담당자가 결정됨.

**최소 버전:** 실제 해외지점 메일에서 branch + system_type을 정확히 추출하고
올바른 Slack 사용자에게 stub 메시지를 보내는 것. 요약의 품질은 2순위.

## Constraints

- 내부 인프라에서 실행 (메일 내용이 외부 SaaS를 통과하는 경우 보안 심사 필요)
- ※ OpenAI API 사용 시 메일 내용이 외부로 전송됨 — 보안팀 사전 확인 필요
- 메일박스: **Gmail (Google Workspace)** — Microsoft 365 아님
- 라우팅 테이블은 비개발자도 관리 가능해야 함

## Premises

1. 핵심 가치 = 누락 없는 전달 (요약 품질보다 우선)
2. 지점 × 시스템 2D 라우팅 테이블 필요. 수동으로 관리되어야 함.
3. LLM 실패 시 fallback: stub 메시지 ("Alert from [branch/system], no summary available") + thread에 원본 plain text 3000자 이내 첨부 (담당자가 원본이 필요한 케이스 대응)
4. 해외지점 메일이 수신되는 메일박스에 접근 가능

※ 전제 #3은 subagent와 이견 있음: subagent는 원본이 너무 길어 Slack에서 비가독적이라고 지적. 담당자 판단으로 원본 유지 결정.

## Cross-Model Perspective

SECOND OPINION (Claude subagent):

> 스팸매직 포인트: 라우팅 매트릭스가 2D이고 수동 관리이므로 시간이 지나면 반드시
> drift됩니다. 파싱을 해결해도 유지보수를 해결 못하면 실패 모드가 "메일 누락"에서
> "잘못된 사람에게 전달"로 이동할 뿐입니다. 설계 우선순위: 라우팅 테이블 staleness를
> 눈에 보이게 하고 수정하기 쉽게 만드는 것.
>
> 프로토타입 제안: Python + Microsoft Graph API + GPT-4o-mini + Slack Incoming Webhook
> + SQLite + Google Sheet 라우팅 테이블. 48시간 빌드.
> 첫 번째 검증 목표: 실제 이메일에서 branch + system_type이 정확히 추출되는가.

## Approaches Considered

### Approach A: No-Code (Zapier/Make)
- Zapier 메일박스 연동 → AI 요약 → Google Sheet 라우팅 → Slack
- Effort: S | Risk: Medium | Completeness: 6/10
- 장점: 빠른 시작, 비개발자 운영 가능
- 단점: 2D 라우팅이 no-code에서 복잡, 외부 SaaS 보안 검토 필요, 비용 확장성 낮음

### Approach B: Python 커스텀 서비스 ✓ (선택)
- Python polling + Microsoft Graph API + GPT-4o-mini + Slack Webhook + SQLite
- Effort: M | Risk: Low | Completeness: 9/10
- 장점: 라우팅 로직 완전 제어, fallback 정확히 설계, 내부 서버 동작, 감사 로그
- 단점: 운영 서버 필요, 코드 유지보수

### Approach C: Microsoft Power Automate
- M365 네이티브: Power Automate + Azure OpenAI + SharePoint 라우팅 리스트 + Slack
- Effort: S-M | Risk: Low | Completeness: 7/10
- 장점: IT 컴플라이언스 친화적, 이미 있는 라이선스 활용
- 단점: 복잡 로직 디버깅 어려움, Azure OpenAI 별도 설정

## Recommended Approach

**Python 커스텀 서비스 (Approach B)**

### 핵심 모듈

**1. 메일 폴러** (내부 polling loop, `time.sleep(120)`)
- **Gmail API** (`google-auth` + `googleapiclient` Python 라이브러리) 로 지정 메일박스 폴링
- 인증: Google Cloud 프로젝트에서 OAuth 2.0 서비스 계정 또는 사용자 OAuth credentials
- 폴링 쿼리: `q="is:unread"` + 특정 발신자 도메인 필터 (해외지점 메일 선별)
- **페이지네이션:** `nextPageToken`을 끝까지 follow (네트워크 단절 후 메일 폭주 시 누락 방지)
- 처리 완료 메일: `UNREAD` 라벨 제거 + 커스텀 라벨 `ROUTER_PROCESSED` 추가
- **프로토타이핑 도구:** `gws` CLI (https://github.com/googleworkspace/cli) — Gmail API를 CLI로 감싼 도구. `gws gmail messages list` 명령으로 로컬 테스트 및 프롬프트 검증에 활용 가능. 프로덕션 폴러는 Python SDK 직접 사용.
- **Gmail API 승인 지연 시 컨틴전시:** Gmail IMAP (앱 비밀번호)로 임시 대체. 인터페이스는 동일하게 설계. LLM API 미확보 시 regex 키워드 매칭으로 임시 분류기 사용.
- **Watchdog:** poll 성공이 6분 이상 없으면 (= polling interval × 3) `#ops-alerts` 채널 경고. watchdog 타이머는 poll 완료 시 리셋 (시작 시 아님).
- **중복 방지 상태머신:** `mail_id` 기준으로 SQLite에 `SEEN → PROCESSING → SENT` 상태 저장.
  - Slack 발송 성공 + DB 상태 `SENT` 업데이트는 단일 트랜잭션으로 처리 (발송 성공 후 DB 쓰기 실패 → 재발송 방지)
  - Slack 발송 실패 시 3회 재시도, 실패 지속 시 `SENT_FALLBACK` 상태로 전환 후 `#ops-alerts` 알림
  - 서비스 재시작 시 `PROCESSING` 상태 메일은 자동 재처리
- **단일 인스턴스 보장:** PID 파일 (`/var/run/mail-router.pid`) 방식. 시작 시 PID 파일 존재하면 기존 프로세스 확인 후 종료 또는 자동 중단. systemd `Restart=on-failure` 사용 시 동시 실행 불가.

**2. 분류기** (LLM prompt)
- GPT-4o-mini에 branch + system_type 추출 요청, `response_format: { type: "json_object" }` 강제
- 파싱 후 `branch`, `system_type` 키 존재 여부 명시적 검증. 키 없으면 오류 로그 후 unknown 처리.
- **실패 시:** `{"branch": "unknown", "system_type": "unknown"}` — unmatched-alerts로 라우팅
- **다국어 대응:** 프롬프트에 "언어 무관하게 branch와 system_type을 대문자 영어로 반환" 명시
- **정확도 게이트:** 배포 전 샘플 메일 20건 수동 검증. 목표 85% 이상 → few-shot 예시 3건 추가 후 95% 달성 예상. 미달 시 프롬프트 재설계 또는 rule-based 보완.

**3. 라우팅 엔진**
- SQLite 라우팅 테이블 스키마:

  | branch (TEXT) | system_type (TEXT) | slack_user_id (TEXT) | channel_id (TEXT) |
  |---|---|---|---|
  | EUROPE | ERP | U1234ABCD | #it-infra-europe |
  | AMERICAS | NETWORK | U5678EFGH | #it-infra-americas |
  | (default) | (default) | — | #unmatched-alerts |

  - `branch`: 지점코드 대문자 (예: EUROPE, AMERICAS, SEA)
  - `system_type`: 시스템 유형 대문자 (예: ERP, NETWORK, SERVER, APP)
  - 매칭: 대소문자 무시, 공백 trim
- **라우팅 테이블 관리:** CSV 파일 → SQLite sync. CSV는 SharePoint 또는 내부 git repo에 보관.
  sync 주기: 서비스 시작 시 + 매 1시간마다 자동. 긴급 변경 시 `SIGHUP` 시그널로 즉시 재로드 (systemd: `kill -HUP <pid>` 또는 `systemctl reload`).
- **정규화:** query time에 `branch.upper().strip()`, `system_type.upper().strip()` 적용. LLM 프롬프트에도 대문자 반환 요청하여 일관성 확보.
- **매핑 없는 경우:** `#unmatched-alerts` 채널 전송 (v1)
- **staleness 감지:** Phase 2로 전면 이관. v1에서는 unmatched 건이 발생할 때마다 `#unmatched-alerts` 기록으로 수동 확인.

**4. Slack 발송**

Fallback 동작 (단일 정의):
- 정상: 메인 메시지 = 3문장 요약 + branch/system 레이블 + 담당자 @멘션 + 수신 타임스탬프 (UTC). thread에 원본 plain text 3000자 이내 첨부 (초과 시 "...[truncated]" 표시).
- LLM 실패: 메인 메시지 = `"[branch/system] Alert — 요약 불가"` + 동일 thread 포맷
- 라우팅 실패: `#unmatched-alerts` 채널에 동일 포맷으로 전송
- **모든 발송:** `mail_received_at` (UTC) + `slack_sent_at` (UTC) SQLite 기록 → 5분 SLA 모니터링. SLA 초과 시 `#ops-alerts` 자동 알림.

**5. 감사 로그**
- 모든 수신 메일의 `mail_id`, `received_at` (UTC) 즉시 기록 (처리 이전) → "누락 0건" 검증 가능
- 일별 수신 건수 vs. 발송 건수 대조 (단일 SQL query): `SELECT COUNT(*) WHERE date=today AND state != 'SENT'`. 불일치 시 `#ops-alerts` 알림. (cron `0 9 * * *` 형태의 단순 스크립트)

## Open Questions

1. **Gmail API 접근 방식:** 서비스 계정(도메인 위임) vs. 사용자 OAuth — Google Workspace admin 협조 필요 여부 확인. 승인 지연 시 IMAP으로 먼저 시작.
2. OpenAI API 키 또는 Azure OpenAI 엔드포인트 확보 가능한가? (메일 내용 외부 전송 승인 포함)
3. 해외지점 메일 언어: 영어 단일인가 다국어인가? (프롬프트 few-shot 예시 설계에 직접 영향)
4. 라우팅 테이블 유지 담당자는 누구인가? 지점/담당자 변경 시 업데이트 프로세스는?
5. Slack Incoming Webhook 또는 Bot Token 설정 권한이 있는가?
6. SQLite 단일 인스턴스 실행 보장: systemd `Restart=on-failure` + 병렬 실행 방지 설정 확인 필요

## Success Criteria

- 해외지점 장애 메일이 수신 후 5분 내에 담당자 Slack에 도달
- 라우팅 정확도 95% 이상 (branch + system_type 분류)
- 메일 누락 0건 (fallback이 항상 작동)
- 2주 운영 후 "이전보다 빠르게 알게 됐다"는 담당자 피드백

## Distribution Plan

- 내부 서버(VM/컨테이너)에서 cron 또는 systemd로 실행
- 코드는 내부 git repo에 관리
- CI/CD: 없어도 됨 (수동 배포로 시작, 안정화 후 파이프라인 추가)

## Dependencies

- **Google Cloud 프로젝트 + Gmail API 활성화** (Google Cloud Console에서 생성, OAuth credentials 발급)
  - 서비스 계정 방식: Google Workspace admin이 도메인 위임(Domain-wide Delegation) 설정 필요
  - 사용자 OAuth 방식: 메일박스 소유자가 앱 승인 1회 필요
- OpenAI API 키 (또는 Azure OpenAI 엔드포인트)
- Slack Incoming Webhook 또는 Slack Bot Token
- 내부 서버 (기존 인프라 활용 가능)
- Node.js 18+ (선택 — `gws` CLI 프로토타이핑용)

## The Assignment

지금 당장 할 한 가지: **지난 2주간 수신된 해외지점 장애 메일 10~20건을 ChatGPT에 붙여넣고
"이 메일에서 branch(지점코드)와 system_type(시스템 유형)을 JSON으로 추출해줘"라고 실행해보세요.**

결과가 85% 이상 정확하면 → 자동화 빌드 시작.
85% 미만이면 → few-shot 예시(정답 2~3건)를 프롬프트에 추가 후 재검증.

자동화 코드 한 줄 없이 가장 중요한 리스크를 제거하는 방법입니다.

## What I noticed about how you think

- "지점 + 시스템 둘 다 복합적으로 다름"이라고 바로 답했습니다. 대부분 "지점별"이나
  "시스템별" 하나만 말합니다. 두 축을 동시에 생각하는 것은 시스템 설계자의 시각입니다.

- fallback에 대해 "원본 메일이 필요한 상황이 있다"고 밀어붙였습니다. 이유가 있는
  거부였고, AI 권고를 그냥 수용하지 않았습니다. 이 설계를 운영할 사람이 맞습니다.

- 문제를 "메일 요약"으로 정의한 것에서 시작했지만 대화를 통해 진짜 문제가
  "라우팅과 누락 방지"라는 것을 빠르게 수용했습니다. 프레임을 바꿀 수 있는 사람입니다.
